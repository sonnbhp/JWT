<ai_system_protocol>
<purpose>
Thiết lập quy trình "Automated Git Workflow" (Quản lý phiên bản rảnh tay) với GitHub: toàn bộ luồng tạo nhánh (branch), commit, push và quản lý mã nguồn được tự động hóa. Người dùng không cần gõ bất kỳ lệnh Git nào. AI sẽ xử lý từ việc soạn commit message chuẩn Conventional Commits đến đẩy code lên GitHub sau khi @A-Test phê duyệt. (Lưu ý: Đây là luồng quản lý git ở local automation, chưa bao gồm server-side CI/CD Pipeline).
</purpose>

<strict_rules>
- TUYỆT ĐỐI KHÔNG PUSH KHI CHƯA QUA TEST: Không bao giờ chạy git push nếu @A-Test chưa xác nhận PASS.
- CONVENTIONAL COMMITS BẮT BUỘC: Mọi commit message phải tuân theo chuẩn `type(scope): description` — không commit bừa bãi.
- TỰ ĐỘNG HOÀN TOÀN: AI phải đề xuất và thực hiện toàn bộ luồng Git mà không cần người dùng gõ lệnh thủ công.
</strict_rules>

<execution_workflow>

### Bước 1: Kết nối và Tạo Repository Tự động (Pre-flight Check)
1. Kiểm tra Git và GitHub CLI đang hoạt động bằng lệnh `git --version` và `gh --version`. 
   - Nếu chưa cài đặt định dạng `Git`, hãy cung cấp đường link tải chính thức (https://git-scm.com/) kèm hướng dẫn cài đặt, hoặc đề xuất lệnh tự chạy (VD: `winget install --id Git.Git -e` trên Windows).
   - Nếu chưa có `gh`, đề xuất lệnh cài đặt tự động (VD: `winget install --id GitHub.cli` trên Windows hoặc `brew install gh` trên Mac).
2. Chạy lệnh `gh auth status` để kiểm tra trạng thái kết nối GitHub.
   - Nếu chưa đăng nhập, AI TUYỆT ĐỐI KHÔNG bắt người dùng gõ lệnh terminal.
   - AI phải tự động chạy ngầm lệnh: `gh auth login --hostname github.com -p https --web`
   - Đọc kết quả từ lệnh trên, trích xuất mã OTP 8 chữ số (VD: `3B78-DC07`) và URL đăng nhập (`https://github.com/login/device`).
   - Cung cấp link và OTP cho người dùng trên khung chat, yêu cầu họ nhấp vào link, nhập mã và Approve. Đợi người dùng báo xong thì mới làm tiếp Bước 3.
3. Chạy `git remote -v` tại thư mục gốc dự án để kiểm tra remote origin đã có chưa.
4. Nếu chưa có remote:
   - AI tự động đọc tên thư mục dự án hiện tại để làm tên repository.
   - Hỏi người dùng muốn tạo repo ở chế độ Public hay Private.
   - Tự động chạy lệnh tạo repo trên GitHub và liên kết với local: 
     `gh repo create [tên-thư-mục] --[public/private] --source=. --remote=origin`
   - Thực hiện khởi tạo và push code hiện tại lên nhánh `main` luôn.

### Bước 2: Thiết lập Git Config Cơ bản
1. Kiểm tra `git config user.name` và `git config user.email`.
2. Nếu chưa có, AI đề xuất:
   ```bash
   git config --global user.name "Tên của bạn"
   git config --global user.email "email@example.com"
   ```
3. Tạo/cập nhật file `.gitignore` phù hợp với dự án (node_modules, .env, logs/).

### Bước 3: Chuẩn hóa Commit Message (Conventional Commits)
Trước khi commit, AI phân tích các file đã thay đổi và tự động soạn message theo chuẩn:

```
<type>(<scope>): <short description>

[optional body]
[optional footer]
```

**Bảng type chuẩn:**
| Type | Dùng khi |
| :--- | :--- |
| `feat` | Thêm tính năng mới |
| `fix` | Sửa lỗi |
| `style` | Thay đổi CSS/UI (không ảnh hưởng logic) |
| `refactor` | Tái cấu trúc code, không thêm tính năng/fix bug |
| `docs` | Cập nhật tài liệu (README, ARCHITECTURE) |
| `chore` | Công việc dọn dẹp, cấu hình |
| `test` | Thêm hoặc sửa test |

**Ví dụ commit message tốt:**
```
feat(profile): add XP system with level progression logic

- calculateXP() based on board size and completion time
- getLevelFromXP() with 10-level threshold system
- addXPToProfile() saves to localStorage automatically
```

### Bước 4: Quy trình Push Tự động (Auto-Push Protocol)
Quy trình này chỉ được kích hoạt sau khi @A-Test ký PASS:

**4a. Tạo Feature Branch (Bắt buộc):**
AI tự động tạo branch mới cho mỗi tính năng/fix thay vì commit thẳng lên `main`:
```
git checkout -b feat/[tên-tính-năng]
# Ví dụ: git checkout -b feat/daily-hack-protocol
```
Quy tắc đặt tên branch theo prefix:
- `feat/` → Tính năng mới
- `fix/` → Sửa lỗi  
- `docs/` → Cập nhật tài liệu
- `refactor/` → Tái cấu trúc

**4b. Commit & Push Branch:**
```
1. git add -A
2. git commit -m "[message chuẩn Conventional Commits]"
3. git push origin feat/[tên-tính-năng]
```

> **⚠️ Lưu ý Shell Compatibility:**
> - **Windows PowerShell**: KHÔNG dùng `&&` để nối lệnh — dùng `;` (semicolon) thay thế.
>   - ✅ Đúng: `git add . ; git commit -m "feat: ..."`
>   - ❌ Sai: `git add . && git commit -m "feat: ..."` → lỗi `ParserError: Invalid statement separator`
> - **Windows CMD / Mac / Linux**: `&&` hoạt động bình thường.
> - Nếu không chắc shell hiện tại, AI phải chạy từng lệnh riêng biệt thay vì nối chuỗi.

**4c. Báo cáo và Chờ Phê duyệt:**
Sau khi push branch xong, AI thông báo rõ ràng cho người dùng:
```
✅ Đã push branch: feat/[tên-tính-năng]
🔗 Link: https://github.com/[user]/[repo]/tree/feat/[tên-tính-năng]

Bạn có muốn merge vào main không? [Trả lời: có/không]
```

**4d. Merge vào main (Chỉ khi người dùng phê duyệt):**
```
git checkout main
git merge feat/[tên-tính-năng] --no-ff
git push origin main
git branch -d feat/[tên-tính-năng]  # Dọn dẹp branch cũ
```

AI báo cáo kết quả merge. Nếu có conflict, AI tự động:
- Chạy `git pull --rebase origin main`
- Giải quyết conflict nếu có thể, hoặc báo cáo rõ ràng để người dùng quyết định.

### Bước 5: Cập nhật RULES.md (Git Governance)
Mở file `RULES.md` và bổ sung quy tắc Git:
- *"Mọi thay đổi code CHỈ được push lên GitHub sau khi @A-Test đóng dấu PASS. AI phải tự động thực hiện commit và push — người dùng không cần gõ lệnh Git."*
- *"Commit message bắt buộc theo chuẩn Conventional Commits. Không chấp nhận message dạng 'fix bug' hay 'update code'."*

### Bước 6: Thông báo hoàn tất
Báo cáo link commit vừa push lên GitHub và nhắn: *"Hệ thống 'Rảnh tay' với GitHub đã hoạt động! Mọi thay đổi được kiểm định bởi @A-Test trước khi lên cloud. Bạn chỉ cần tập trung vào ý tưởng."*
